  ---
  ICC 类别表现分析

  当前结果

  17个benign案例全部应为 exploitable=True（benign = 有漏洞的受害者app）
  - ✅ TP: 3/17（18% recall）
  - ❌ FN: 14/17（82% miss rate）

  ---
  根本原因分析

  根因 1：Validation Agent 从不查询 CrossPath 节点（最严重）

  ICC Bridge 正确写入了 CrossPath 节点（7个案例有 cross_paths ≥ 1），但
  validation_agent 的 ReAct 循环从未查询 MATCH 
  (cp:CrossPath)。它固定在3个查询模式上：
  1. Component {exported:true} → 无结果
  2. EVIDENCED_BY SASTFinding WHERE fused_score > 0.5 → 无结果
  3. MATCH (sf:SASTFinding) WHERE fused_score > 0.5 → 无结果

  然后直接输出 exploitable=False。数据明明在图里，但 agent 不去看。

  根因 2：manifest_agent 是 ContentProvider-centric 的

  manifest_agent 只对 ContentProvider 写 vuln_description，且只检查
  exported=true。对 ICC 特有的漏洞属性完全不处理：

  ┌────────────────────────────┬─────────────────────────┬─────────────────┐
  │          漏洞类型          │        关键属性         │ manifest_agent  │
  │                            │                         │    是否检测     │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ TaskAffinity-ActivityHijac │ android:taskAffinity    │ ❌ 不检测       │
  │ k                          │ 与其他app相同           │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ HighPriority-ActivityHijac │ intent-filter           │ ❌ 不检测       │
  │ k                          │ android:priority="5"    │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ DynamicRegBroadcastReceive │ smali中                 │ ❌              │
  │ r                          │ registerReceiver()      │ 不在manifest里  │
  │                            │ 无权限                  │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ EmptyPendingIntent         │ PendingIntent.get*(null │ ❌ 不检测       │
  │                            │ )                       │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ ImplicitPendingIntent      │ PendingIntent +         │ ❌ 不检测       │
  │                            │ 隐式Intent              │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ StickyBroadcast            │ sendStickyBroadcast +   │ ❌ 不检测       │
  │                            │ BROADCAST_STICKY 权限   │                 │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ OrderedBroadcast           │ exported=true receiver  │ ❌ 漏写 vuln_de │
  │                            │ + NEW_OUTGOING_CALL     │ scription       │
  ├────────────────────────────┼─────────────────────────┼─────────────────┤
  │ FragmentInjection          │ PreferenceActivity + 无 │ ❌ 无 Activity  │
  │                            │  isValidFragment        │ 导出            │
  └────────────────────────────┴─────────────────────────┴─────────────────┘

  根因 3：ICC 漏洞本质是结构性，不是数据流

  当前 pipeline 的核心是：source → taint propagation → sink。但 14个 FN
  案例中，大多数不需要数据流：

  - TaskAffinity（4个案例）：纯 manifest 属性漏洞，不存在 source/sink
  - Dynamic Broadcast：配置漏洞，registerReceiver 没有权限参数
  - PendingIntent（2个案例）：API 使用模式漏洞，不是数据流
  - HighPriority：Intent 竞争漏洞
  - Sticky/Ordered Broadcast：broadcast 机制滥用

  这些 sources=0, sinks=0 是预期的，不是 bug，而是这类漏洞本来就不需要 taint 
  分析。

  根因 4：SAST ToolPrior 对 ICC 模式覆盖弱

  所有 FN 案例的 SAST aligned findings 的 fused_score 均 < 0.5（阈值），导致
  validation_agent 不信任。主因是：11 款 SAST 工具里很少有专门检测
  TaskAffinity、PendingIntent、Dynamic Receiver 的专项能力。

  ---
  优化建议（按优先级）

  优化 1：立即修复 - Validation Agent 必须首先查询 CrossPath【预期 +3~4 TP】

  在 validation_agent 的 system prompt 中增加固定的预置查询，在 ReAct
  循环开始前自动执行：

  # validation_agent.py - 在 agent_loop 之前先做预查询
  prefetch_queries = [
      "MATCH (cp:CrossPath) RETURN cp.id, cp.channel_type, cp.target, 
  cp.confidence",
      "MATCH (c:Component) WHERE c.vuln_description IS NOT NULL RETURN c.name, 
  c.exported, c.vuln_description",
  ]

  或者更简单：在 user_msg 的初始提示里直接把 CrossPath 数据 inject
  进去，而不是靠 LLM 自己去查。

  优化 2：manifest_agent 增加 ICC 结构性漏洞检测【预期 +5~6 TP】

  扩展 manifest_agent 的分析范围，超出 ContentProvider：

  # 检测 taskAffinity 错误配置
  MATCH (c:Component) WHERE c.taskAffinity IS NOT NULL
    AND NOT c.taskAffinity STARTS WITH app_package RETURN c

  # 检测 intent-filter priority > 0
  MATCH (c:Component)-[:HAS_INTENT_FILTER]->(f) WHERE f.priority > 0 RETURN c,
  f.priority

  # 检测带 intent-filter 但无权限的 receiver/service
  MATCH (c:Component) WHERE c.type IN ['receiver','service']
    AND c.exported = true AND c.permission IS NULL RETURN c

  同时，将这些检测结果写入 vuln_description，让 validation_agent 可以读到。

  优化 3：新增 ICC 结构分析节点（icc_pattern_node）【预期 +6~8 TP】

  在 manifest_agent 之后、taint_agent 之前插入新节点，专门做 ICC
  结构性模式检测：

  manifest_agent → icc_pattern_node(NEW) → ui_semantic_agent → sast_prior_node →
   ...

  该节点检测（全部程序化，无 LLM）：

  ┌───────────────────────────────────────────┬──────────────────────────────┐
  │                 检测目标                  │           检测方法           │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ android:taskAffinity 与其他 app 共享      │ 解析 manifest XML            │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ intent-filter priority > 0                │ 解析 manifest XML            │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ registerReceiver() 无权限参数             │ smali grep registerReceiver  │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ PendingIntent.get*(null/empty intent)     │ smali grep PendingIntent API │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ sendStickyBroadcast                       │ smali grep                   │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ PreferenceActivity 未覆盖 isValidFragment │ smali grep                   │
  ├───────────────────────────────────────────┼──────────────────────────────┤
  │ exported receiver + system action 无权限  │ manifest XML                 │
  └───────────────────────────────────────────┴──────────────────────────────┘

  检测结果以 IccPattern 节点写入 Neo4j，或直接在 Component 上加 icc_vuln_pattern
   属性。

  优化 4：Validation Agent 降低 ICC CrossPath 的证明门槛

  对 ICC 类别专用的判断逻辑：如果 CrossPath 存在 + channel_type
  是高危类型，直接升级置信度，无需等 SAST fused_score > 0.5：

  HIGH_RISK_CHANNELS = {"DYNAMIC_BROADCAST", "IMPLICIT_INTENT",
  "CONTENT_PROVIDER_EXPORT"}
  # CrossPath confidence >= 0.4 + channel in HIGH_RISK_CHANNELS → 
  exploitable=True

  优化 5：修复 OrderedBroadcast 案例（最近的快速 TP）

  OrderedBroadcast-DataInjection 的 manifest 里有 exported=true 的 receiver 接收
   NEW_OUTGOING_CALL，但 manifest_agent 没有写 vuln_description。检查
  manifest_agent 是否遗漏了 receiver/service 类型。

  ---
  优化路线图

  优先级1（立即）:
    - 修复 validation_agent 预置 CrossPath 查询  → +3-4 TP
    - 修复 manifest_agent 覆盖 receiver/service → +1-2 TP

  优先级2（中期）:
    - 新增 icc_pattern_node                    → +5-6 TP
    - validation_agent ICC 专项判断逻辑         → +1-2 TP

  预期总效果: 3/17 → 12-14/17 recall (70-82%)

  最核心的问题是：pipeline 的整个验证逻辑假设漏洞 = taint flow，但 ICC 中 >60% 
  的案例是结构性配置漏洞，根本不走 source→sink 路径。
  这是设计层面的盲区，需要引入专门的结构性 ICC 模式检测。

  做了真正的过程间 taint 分析，APKHunt 只是 grep 
  了一下字符串。两者的可信度一样吗？

  不一样。 但如果你不知道每个工具是"怎么检测的"，你就只能把所有报告平等对待，无
  法区分深浅。

  ToolPrior 就是为了解决这个问题：它记录了每个工具每个检测能力的"分析深度"，这样
  在融合时就能做到按能力加权，而不是平等投票。

  ---
  Schema 的核心结构（三层）

  ToolPrior（工具级）
    ├── tool_name: "JAADAS"
    ├── taint_relevance: HIGH          ← 整体与 taint 的相关性
    ├── confidence_weight: 0.5         ← 初始可信度（后续反馈闭环更新）
    │
    └── capabilities: [                ← 能力列表（核心）
          DetectionCapability（能力级）
            ├── category: "sensitive data flow"
            ├── cwe_ids: ["CWE-200"]
            ├── signal_polarity: VULNERABILITY     ←
  报的是漏洞？还是加固？还是观察？
            ├── taint_relevance: HIGH              ← 这个具体能力对 taint 的价值
            ├── fp_risk: MEDIUM                    ← 误报风险
            │
            └── detection_methods: [               ←
  怎么检测的（可以有多种方法组合）
                  DetectionMethod
                    ├── family: FLOWDROID_TAINT    ← 检测机制族
                    ├── depth: TAINT_ANALYSIS      ← 分析深度
                    └── target_artifact: "jimple"  ← 分析对象
                ]
        ]

  ---
  6 个关键设计决策（创新点）

  1. DetectionMethod 是结构体，不是简单枚举

  朴素做法：给每个检测能力贴一个标签，比如 detection_method = "regex"。

  问题：很多工具的一个检测能力混合使用多种方法。比如 AndroBugs 检测不安全
  SSL：先查"类是否实现了 X509TrustManager 接口"（类层级检查），再查"方法体是不是
  空的"（语义启发式），再追溯"谁调用了它"（调用者追踪）。三步组合，不是一个 enum
   能描述的。

  所以 detection_methods 是一个 list[DetectionMethod]，每个 method 有
  family（做了什么）+ depth（做到多深）。

  2. AnalysisDepth — 8 级分析深度

  这是融合评分时最关键的维度：

  syntax_only        → 纯字符串匹配（APKHunt grep 代码）          权重最低
  manifest_parse     → 解析 XML 属性
  api_presence       → 检测 API 调用存在，但不追踪数据流
  constant_backtrack → 回溯常量参数（如加密密钥是不是硬编码的）
  intra_procedural   → 方法内部的流/状态追踪
  inter_procedural   → 跨方法/跨类的分析
  taint_analysis     → 完整的 source→sink 污点分析（FlowDroid）  权重最高
  external_result_ingest → 消费其他工具的结果

  同一个漏洞，JAADAS 用 taint_analysis 查出来的 vs APKHunt 用 syntax_only grep
  出来的，融合时得分差距很大。这就是"能力感知"的含义。

  3. SignalPolarity — 区分漏洞/加固/观察

  不是所有 SAST 报告都在说"有漏洞"。比如：
  - MobSF 报 "network security config found" →
  这是说防护已存在（HARDENING_PRESENT）
  - AndroBugs 报 "No SSL certificate pinning" →
  这是说防护缺失（HARDENING_MISSING）
  - Trueseeing 报 "library: okhttp 3.x detected" → 这是纯观察（OBSERVATION）

  如果不区分，把加固信号当漏洞信号融合，下游推理就被噪声干扰了。

  4. 不设强制 confidence_score

  调研了 11 个工具，没有一个有校准过的置信度分数。Marvin 有个 ad hoc 的
  0.5/0.9，JAADAS 有个启发式权重，但都不是统计意义上的置信度。

  所以 ToolPrior 不在 finding 级别设 confidence，而是在工具级别设
  confidence_weight（初始 0.5），靠后续的反馈闭环（Phase
  3）从实验结果中学习更新。

  5. taint_relevance 双层设置

  工具级别和能力级别各有一个 taint_relevance：
  - AndroBugs 工具整体对 taint 相关性 = LOW
  - 但 AndroBugs 的 "exported component" 能力 → 对 taint 的 entry point 有用 =
  MEDIUM
  - AndroBugs 的 "Base64 string" 能力 → 对 taint 无直接价值 = LOW

  这让在线融合模块在写 sast_taint_hint 到 Method
  节点时，可以按能力级别精细判断。

  6. fp_risk + known_quirks 追踪实现可靠性

  多个工具有已知的实现 bug（DroidStatX 只读文件第一行、Marvin
  类名错位、Trueseeing 的 .lower 未调用）。这些直接影响输出可信度。fp_risk 和
  known_quirks 把这些信息结构化记录下来，让融合评分时能打折扣。

  ---
  在线融合时怎么用？

  fused_score = capability_strength × evidence_strength × reliability ×
  severity_weight

  其中 capability_strength 就是从 DetectionMethod.depth 映射来的：

  DEPTH_SCORES = {
      "syntax_only": 0.2,
      "manifest_parse": 0.35,
      "api_presence": 0.4,
      "constant_backtrack": 0.55,
      "intra_procedural": 0.65,
      "inter_procedural": 0.8,
      "taint_analysis": 0.95,
  }

  所以同一个 CWE 的 finding，JAADAS（taint_analysis, 0.95）报的就是比 
  APKHunt（syntax_only, 0.2）报的得分高 4-5 倍。 这就是 ToolPrior Schema
  的核心价值。

❯ SAST Finding是节点吗？                                                        

⏺ 是的，SASTFinding 是 Neo4j 里的一个独立节点类型。

  它代表的是：11 个 SAST 工具经过能力感知融合后，对齐到图上的一条具体发现。

  (SASTFinding {
      id: "mobsf_task_affinity_set_abc123",
      tool_name: "mobsf",
      title: "task_affinity_set",
      fused_score: 0.42,
      capability_analysis_depth: "manifest_parse",
      alignment_status: "aligned",        ← 已对齐到图上某个节点
      matched_node_type: "Component",
      matched_node_id: "ImageEditor"
  })

  它通过两条溯源边连接到图中已有的程序结构节点：

  (SASTFinding)──IMPLICATES──→(Component "ImageEditor")
  (Component "ImageEditor")──EVIDENCED_BY──→(SASTFinding)

  为什么要做成独立节点而不是直接在 Component/Method 上加属性？

  因为一个 Component 可能被多个工具的多条 finding 指向。如果只用属性，就丢失了"
  哪个工具、用什么方法、查到了什么"的详细信息。做成节点后，validation_agent
  可以查：

  -- 查某个组件被哪些工具标记过，各自得分多少
  MATCH (c:Component {name:"ImageEditor"})-[:EVIDENCED_BY]->(sf:SASTFinding)
  RETURN sf.tool_name, sf.title, sf.fused_score, sf.capability_analysis_depth

  这就保留了完整的证据链，让 LLM 最终裁定时能看到"3 个工具都报了这个组件，其中
  JAADAS 用 taint_analysis 查的（高分），APKHunt 用 regex 查的（低分）"。

  ① manifest_agent — "门卫，看谁对外暴露了"

  输入：AndroidManifest.xml
  做什么：用 LLM 分析哪些组件 exported=true，有没有权限保护缺失。重点查
  ContentProvider 的路径权限是否有漏洞（如只保护了 /user，根路径 / 没保护）。
  写回图：在 Component 节点上写 vuln_description、root_path_protected
  作用：确定外部攻击者能"摸到"哪些入口

  ---
  ② ui_semantic_agent — "看界面，哪些输入框是敏感的"

  输入：res/layout/*.xml 布局文件
  做什么：扫描所有 EditText 等输入控件，用
  LLM（或规则回退）判断每个控件的敏感度：是密码框？手机号？还是普通文本？
  写回图：创建 UIView 节点（含 sensitivity_label），连 READS_UI 和 HAS_UI 边
  作用：为 taint_agent 提供额外的 source 候选（高敏感 UI 输入 → 可能的污点源头）

  ---
  ③ sast_prior_node — "11 个工具的报告融合打分，写证据到图上"

  输入：11 个 SAST 工具的扫描报告 + 11 个 ToolPrior JSON
  做什么（全程序化，无 LLM）：
  1. 解析：11 种格式的报告 → 统一的 NormalizedFinding
  2. 融合：每条 finding 匹配到对应工具的对应 capability，用分析深度计算
  fused_score
  3. 对齐：把 finding 匹配到图上已有的 Component/Method 节点
  4. 写入：创建 SASTFinding 节点 + IMPLICATES/EVIDENCED_BY 溯源边

  taint_agent 参考

  作用：把 11 个工具的集体智慧注入图中，但只作为提示，不直接下结论

  ④ taint_agent — "标记污点源头和泄露出口"

  输入：所有 smali 文件 + 图上的 SAST hint
  做什么（两层）：
  - 第一层（规则，置信度 0.95）：扫描 smali 的 invoke-* 指令，和内置的 14 个
  source 黑名单（getDeviceId、getLatitude 等）+ 14 个 sink
  黑名单（sendTextMessage、Log.d 等）取交集

  LLM，让它从中再选出额外的 source/sink。LLM 同时能看到图上的 SAST hint

  写回图：在 Method 节点上写
  taint_role=source/sink、confidence、inference_source
  输出文件：SourcesAndSinks.txt（SuSi XML 格式，给 FlowDroid 用）
  作用：确定"敏感数据从哪来（source）、往哪去（sink）"

  ---
  ⑤ semantic_agent — "深度代码理解（可选）"

  触发条件：taint_agent 置信度 < 0.7 时才触发
  做什么：用 LLM 更深入地阅读 smali + java 代码，重新理解数据流逻辑
  写回图：更新 Method 的 confidence
  作用：对不确定的标注做二次确认

  ---
  ⑥ flowdroid_node — "追踪组件内的数据流路径"

  输入：APK 文件 + SourcesAndSinks.txt
  做什么：调用 FlowDroid 工具，自动寻找从 source 到 sink

  source/sink 生成合成回退路径
  写回图：创建 IntraPath 节点（组件内污点路径）
  作用：回答"数据是否真的从 source 流到了 sink"（Layer A：组件内）

  ---
  ⑦ icc_bridge_node — "跨组件攻击路径分析"

  输入：图上已有的 IntraPath + Component + Method + smali
  做什么：检测 6 种跨组件通信模式：

  ┌──────────────────────┬────────────────────────────────────┐
  │         模式         │                场景                │
  ├──────────────────────┼────────────────────────────────────┤
  │ ContentProvider 导出 │ 外部 App 直接 query 拿数据         │
  ├──────────────────────┼────────────────────────────────────┤
  │ 静态字段共享         │ 组件 A 写静态变量，组件 B 读       │
  ├──────────────────────┼────────────────────────────────────┤
  │ 动态广播             │ sendBroadcast → onReceive          │
  ├──────────────────────┼────────────────────────────────────┤
  │ setResult            │ Activity 通过 result 回传数据      │
  ├──────────────────────┼────────────────────────────────────┤
  │ 显式 Intent          │ startActivity + putExtra 指定目标  │
  ├──────────────────────┼────────────────────────────────────┤
  │ 隐式 Intent          │ 只有 action，按 intent-filter 匹配 │
  └──────────────────────┴────────────────────────────────────┘

  写回图：创建 CrossPath 节点 + ICC 相关边（INTENT_SENDS_TO、BROADCAST_SENDS
  等）
  作用：把组件内路径升级为跨组件攻击链（Layer B：跨组件）

  ---
  ⑧ validation_agent — "最终裁判，有没有漏洞？"

  输入：Neo4j 图中的所有证据
  做什么：用 LLM + ReAct 循环（最多 5 轮），自主构造 Cypher 查询从图中搜集证据：
  - 查导出组件 + vuln_description

  ---
  ⑨ report_generator — "写报告"

  输入：pipeline 共享 state 中所有 agent 的结果
  输出：report.md（人可读的漏洞报告）+ state_dump.json（完整状态快照）

⏺ 从我们之前的 ICC 深度分析中，14 个 FN 的 4 个根因是：

  根因 1：Validation Agent 不查 CrossPath 节点（最致命）

  ICC Bridge 明明已经在图里写了 CrossPath 节点（7 个 FN 案例都有 cross_paths ≥ 
  1），但 validation_agent 的 LLM ReAct 循环从来不查它。5 轮查询全部围绕：

  1. MATCH (c:Component {exported:true}) → 无结果
  2. MATCH ... EVIDENCED_BY SASTFinding WHERE fused_score > 0.5 → 无结果
  3. MATCH (sf:SASTFinding) WHERE fused_score > 0.5 → 无结果

  然后就放弃了，输出 exploitable=False。数据在图里，但 agent 不知道要去看。

  根因 2：manifest_agent 视角太窄

  manifest_agent 只关心 ContentProvider 的 exported=true + 路径权限。对 ICC
  类别里真正关键的漏洞属性完全不检测：

  - TaskAffinity（4 个案例）：android:taskAffinity 设成了跟其他 app 一样的值 →
  可被劫持。manifest_agent 不看这个属性
  - intent-filter priority（1 个案例）：android:priority="5" →
  攻击者用更高优先级抢先响应。不检测
  - 动态 BroadcastReceiver（1 个案例）：smali 里 registerReceiver()
  没带权限参数，但这东西根本不在 manifest 里，manifest_agent 看不到
  - exported receiver（1 个案例，OrderedBroadcast）：manifest 里有 exported=true
   的 receiver 接收 NEW_OUTGOING_CALL，但 manifest_agent 没给它写
  vuln_description

  根因 3：ICC 漏洞本质是结构性的，不是数据流

  这是设计层面的盲区。当前 pipeline 的核心假设是"漏洞 = source → taint
  propagation → sink"。但 14 个 FN 中：

  - 12 个案例 sources=0 — taint_agent 找不到 source，因为本来就没有数据流
  - TaskAffinity 是纯 manifest 属性错误
  - PendingIntent 是 API 使用模式问题（PendingIntent.get*(null)）
  - StickyBroadcast 是 broadcast 机制滥用
  - FragmentInjection 是 PreferenceActivity 没覆盖 isValidFragment
  - UnhandledException 是异常处理缺失导致 DoS

  这些漏洞不存在 source→sink 数据流，taint 分析对它们结构性失效。

  根因 4：SAST ToolPrior 对 ICC 模式覆盖弱

  所有 14 个 FN 案例里，SAST 对齐后的 fused_score 全部 < 0.5（validation_agent
  的查询阈值）。原因是 11 款工具里很少有专门检测
  TaskAffinity、PendingIntent、动态 Receiver 的专项能力。即使 Marvin 能检测
  UNPROTECTED_DYNAMICALLY_REGISTERED_RECEIVER，MobSF 能检测
  task_affinity_set，但它们的分析深度都是 manifest_parse 或
  syntax_only，融合后得分上不去。

  ---
  4 个根因的关系

  根因3（ICC是结构性漏洞）── 根本原因，pipeline设计假设不匹配
    ↓ 导致
  根因2（manifest_agent不检测ICC属性）── 入口就漏了
    ↓ 导致
  根因4（SAST对ICC覆盖弱）── 补救手段也弱
    ↓ 导致
  根因1（validation_agent不查CrossPath）── 最后一道防线也没兜住
    ↓
  exploitable=False（FN）

  根因 3 是根本性的，但根因 1 是最容易修的——只要让 validation_agent 查一下
  CrossPath，就能立刻捞回 3-4 个 TP。

  ICC 是什么？

  ICC = Inter-Component Communication（组件间通信）。Ghera 里 ICC 类别的 17
  个漏洞涵盖：

  - TaskAffinity 劫持（4个）— 纯 manifest 属性配置错误

  去测它们，本身就不是完全对口的。

  icc_bridge_node：它基于 smali 的字符串匹配和 manifest 的 intent-filter 做 ICC
  模式检测。能检测到一些 CrossPath，但：
  1. 置信度偏低（0.4-0.65）
  2. validation_agent 不查 CrossPath，所以即使检测到了也没用
  3. ICC 的结构性漏洞（taskAffinity 等）它也检测不了

  本质问题：这两个节点在当前实验中基本是空转的。FlowDroid 跑不起来 → synthetic
  只是占位符 → icc_bridge 基于占位符做升级没有意义 → validation_agent

  但这不代表它们设计有问题——如果 FlowDroid 能正常工作（修好
  classpath），产出真实的 IntraPath，那 icc_bridge 的 ContentProvider

  所以下一步有两个选择：
  1. 修好 FlowDroid 环境（Soot classpath），让 Layer A 真正跑起来
  2. 绕过 FlowDroid，让 LLM 直接在图上做 taint 推理（这就是你之前 v2
  的思路，也更符合论文的创新点）

❯ 什么是ICC漏洞                                                                 

⏺ ICC = Inter-Component Communication，组件间通信。

  Android 应用由多个组件（Activity、Service、BroadcastReceiver、ContentProvider
  ）组成，它们之间不能直接调用，必须通过 Intent、ContentResolver、Broadcast
  等机制通信。ICC 漏洞就是这些通信机制中存在的安全问题。

  常见的 ICC 漏洞类型：

  ┌──────────────────┬──────────────────────────┬───────────────────────────┐
  │       类型       │           原理           │           例子            │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ 组件导出未保护   │ 组件 exported=true       │ 恶意 App 直接调用你的     │
  │                  │ 但没设权限               │ Service 执行敏感操作      │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ 隐式 Intent 劫持 │ 用隐式 Intent            │ 攻击者注册同名 action 的  │
  │                  │ 发送敏感数据             │ Activity 抢先接收         │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ PendingIntent    │ PendingIntent 用了空     │ 接收方可以任意填充 Intent │
  │ 劫持             │ Intent                   │  内容                     │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ ContentProvider  │ Provider                 │ 恶意 App 通过 content://  │
  │ 泄露             │ 导出且权限保护不全       │ URI 读取用户数据          │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ Broadcast 劫持   │ 发送广播没限制接收者     │ 恶意 App 注册 Receiver    │
  │                  │                          │ 窃听广播中的数据          │
  ├──────────────────┼──────────────────────────┼───────────────────────────┤
  │ TaskAffinity     │ Activity 的 taskAffinity │ 恶意 App 创建同 affinity  │
  │ 劫持             │  可被利用                │ 的 Task 进行钓鱼          │
  └──────────────────┴──────────────────────────┴───────────────────────────┘

  跟 taint 分析的关系：ICC 漏洞中只有一部分涉及数据流（比如通过 Intent
  传递敏感数据、ContentProvider
  泄露数据），另一大部分是纯配置/结构性问题（TaskAffinity、PendingIntent 空
  Intent），不需要 source→sink 分析。我们的 pipeline
  对前者有效，对后者结构性失效。

❯ CheckValidity-InformationExposure和WebView-CookieOverwrite分别是什么类型的漏  
洞，描述一下。                                                                  

⏺ Searched for 2 patterns, read 2 files (ctrl+o to expand)

⏺ CheckValidity-InformationExposure（Networking 类别）

  漏洞：自定义 X509TrustManager，checkServerTrusted()
  方法体为空——不做任何证书校验就放行。

  public void checkServerTrusted(X509Certificate[] chain, String authType) 
      throws CertificateException {
      // 空的！什么都不检查，直接信任任何证书
  }

  后果：App 的 HTTPS 连接接受任何证书（包括攻击者自签的），中间人（MITM）可以窃
  听/篡改全部网络通信。

  CWE：CWE-295（不当的证书验证）

  为什么我们检测不到：这是一个API 误用型漏洞——实现了安全接口但方法体为空。没有
  source→sink 数据流，没有导出组件，SAST 工具虽然能报（如
  JAADAS、AndroBugs），但 fused_score 不够高。pipeline 需要"理解方法体为空 =
  安全检查被绕过"，这属于代码语义分析能力。

  ---
  WebView-CookieOverwrite（Web 类别）

  漏洞：WebViewActivity 从 Intent 中读 URL 然后直接 loadUrl()，且开启了
  JavaScript：

  webSettings.setJavaScriptEnabled(true);
  webView.loadUrl(getIntent().getStringExtra("url"));  // 外部可控的 URL！

  后果：如果攻击者能向这个 Activity 发 Intent（虽然 exported=false，但同 App
  内其他组件可以调用），就能让 WebView 加载任意 URL，配合
  setJavaScriptEnabled(true) + CookieManager，攻击者可以覆盖/窃取 Cookie。

  CWE：CWE-829（不可信来源的功能包含）

  为什么我们检测不到：这里其实有一条 taint 
  flow（getIntent().getStringExtra("url") → loadUrl()），但我们的 taint_agent 的
   source 黑名单里没有 getStringExtra，sink 黑名单里没有 loadUrl。加上
  exported=false，manifest_agent 不报告，validation_agent 就判了无漏洞。如果扩展
   source/sink 规则库，这个案例有可能变 TP。

❯ 我们的创新点是啥？                                                            

⏺ 结合整个研究历程，核心创新点有三个：

  C1：Tool Prior — SAST 工具能力画像

  不是简单地"把多个工具结果投票合并"，而是给每个工具的每个检测能力建了结构化档案
  ，记录它怎么检测的、分析多深、误报风险多大。

  关键区别：
  - 传统做法：3 个工具都报了同一个漏洞 → 可信度高（平等投票）
  - 我们的做法：JAADAS 用 taint_analysis 报的 vs APKHunt 用 regex grep 报的 →
  前者得分是后者的 4-5 倍（能力感知加权）

  融合公式 fused_score = capability_strength × evidence_strength × reliability ×
   severity_weight 中，capability_strength 来自 ToolPrior 里的
  AnalysisDepth，这是区别于现有工作的核心。

  C2：图引导的 LLM 污点推理（Graph-Guided LLM Taint Reasoning）

  LLM 不是直接读代码猜漏洞，而是在 Neo4j 程序图上做结构化推理。

  - HPG（异构程序图）提供精确的程序结构：组件、方法、调用关系、UI 控件、权限配置
  - SAST 融合结果作为图属性注入（sast_taint_hint、sast_fused_score）
  - LLM agent 通过 Cypher 查询图来搜集证据，而非直接读源码
  - 最终裁定基于 GraphRAG（Text2Cypher），有可追溯的证据链

  核心设计原则："SAST enriches, LLM decides"——SAST 只写提示和证据，LLM
  自主决策。溯源边 vs 结构边的分离确保 SAST 不污染图拓扑。

  C3：三层渐进式检测架构

  Layer A（组件内）: taint_agent + FlowDroid → IntraPath
  Layer B（跨组件）: icc_bridge 升级 → CrossPath
  Layer C（GraphRAG 验证）: validation_agent 综合裁定

  每一层在同一张图上累积证据，逐层升级攻击链的完整度。这个分层设计让系统可以
  graceful degradation——即使 FlowDroid 失败（Layer A），ICC Bridge 和 Validation
   Agent 仍可基于图属性和 SAST 证据工作。
