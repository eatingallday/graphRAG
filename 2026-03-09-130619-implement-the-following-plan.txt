│ 三层污点引擎（论文核心架构）                                                 │
│                                                                              │
│ APK                                                                          │
│  │                                                                           │
│  ▼                                                                           │
│ [LLM Agent 推断] source/sink → SuSi XML（供 FlowDroid 使用）                 │
│  │                                                                           │
│  ▼                                                                           │
│ [Layer A] FlowDroid 组件内污点传播                                           │
│          - 输入：APK + SuSi XML + android.jar                                │
│          - 范围：单组件内（ContentProvider.query() 内部）                    │
│          - 输出：IntraPath 列表（source→sink，含 URI 参数到 Cursor 返回值）  │
│          - 写入 Neo4j：(:IntraPath) 节点 +                                   │
│ (:Method)-[:HAS_INTRA_PATH]->(:IntraPath)                                    │
│  │                                                                           │
│  ▼                                                                           │
│ [Layer B] 自定义 ICC Bridge（Python）                                        │
│          - 输入：Neo4j 中的 IntraPath + Component 节点（exported=true）      │
│          - 逻辑：若 ContentProvider exported，则 IntraPath 自动升级为        │
│ CrossPath                                                                    │
│          - 对 Intent 型 ICC：连接 startActivity/sendBroadcast →              │
│ onReceive/onCreate                                                           │
│          - 输出：CrossPath（跨组件完整路径）                                 │
│          - 写入 Neo4j：(:CrossPath) 节点 + 置信度                            │
│  │                                                                           │
│  ▼                                                                           │
│ [Layer C] GraphRAG Validation（Path Validation Agent）                       │
│          - 输入：Neo4j 中所有 CrossPath                                      │
│          - 方式：Text2Cypher 自然语言查询验证可行性                          │
│          - 输出：最终验证结论 + 证据链                                       │
│                                                                              │
│ 分层优势：FlowDroid 负责形式化精确的组件内分析，LLM                          │
│ 负责跨组件语义理解，GraphRAG 负责整体路径可行性验证，三者互补。              │

│ import xml.etree.ElementTree as ET                                           │
│                                                                              │
│ def parse_flowdroid_output(xml_path: str) -> list[dict]:                     │
│     tree = ET.parse(xml_path)                                                │
│     results = []                                                             │
│     for flow in tree.findall('.//Result'):                                   │

│             "source": flow.find('Source').get('Statement'),                  │
│             "sink":   flow.find('Sink').get('Statement'),                    │

│     return results                                                           │

│ STEP 1：初始化 Neo4j HPG（最小图）                                           │
│                                                                              │
│ 文件：hpg/builder.py                                                         │
│                                                                              │
│ HPG 以方法为核心节点（与 FlowDroid 输出对齐，支持精确 source/sink 映射和     │
│ GraphRAG 查询）。                                                            │
│                                                                              │
│ 从 Manifest + Smali 静态构建：                                               │
│                                                                              │
│ # ─── 组件节点（粗粒度，作为方法的容器）                                     │
│ # (:Component {name:"UserDetailsContentProvider", type:"provider",           │
│ #              exported:true, authority:"edu.ksu.cs.benign.userdetails"})    │
│ # (:Component {name:"MainActivity", type:"activity", exported:true})         │
│ # (:Component {name:"UserDetailsActivity", type:"activity", exported:false}) │
│                                                                              │
│ # ─── 方法节点（细粒度，HPG 核心）每个应用方法一个节点                       │
│ # (:Method {sig:"<edu.ksu.cs.benign.provider.UserDetailsContentProvider:     │
│ android.database.Cursor query(android.net.Uri,java.lang.String[],java.lang.S │
│ tring,java.lang.String[],java.lang.String)>",                                │
│ #           name:"query", class:"UserDetailsContentProvider",                │
│ #           is_entry:true, taint_role:"unknown", confidence:0.0})            │
│ # (:Method {sig:"<edu.ksu.cs.benign.provider.UserDetailsContentProvider:     │
│ boolean onCreate()>", ...})                                                  │
│ # (:Method {sig:"<edu.ksu.cs.benign.MainActivity: void                       │
│ onCreate(android.os.Bundle)>", ...})                                         │
│ # (:Method {sig:"<edu.ksu.cs.benign.UserDetailsActivity: void                │
│ onCreate(android.os.Bundle)>", ...})                                         │
│ # ... （此 APK 共 ~15 个应用方法节点）                                       │
│                                                                              │
│ # ─── 权限 / 路径权限节点                                                    │
│ # (:Permission {name:"edu.ksu.cs.benign.permission.internalRead",            │
│ protectionLevel:"dangerous"})                                                │
│ # (:PathPermission {pathPrefix:"/user", readPermission:"...internalRead"})   │
│                                                                              │
│ # ─── 字符串常量节点（与方法关联）                                           │
│ # (:StringConst {value:"/user/ssn",     sensitivity:"HIGH"})                 │
│ # (:StringConst {value:"/user/address", sensitivity:"MEDIUM"})               │
│ # (:StringConst {value:"/user/school",  sensitivity:"LOW"})                  │
│                                                                              │
│ # ─── 边                                                                     │
│ # (Component:Provider)-[:CONTAINS]->(Method:query)       ← 方法归属          │
│ # (Component:Provider)-[:REQUIRES_PERM]->(Permission)    ← 权限要求          │
│ # (Component:Provider)-[:HAS_PATH_PERMISSION]->(PathPermission)              │
│ # (Method:query)-[:ACCESSES]->(StringConst:ssn)          ← 初始 pending，由  │
│ Taint Agent 写入                                                             │
│ # (Method:query)-[:DIRECT_CALL]->(Method:其他内部调用)   ← 由 Smali 解析填充 │
│                                                                              │
│ 为什么方法级别：                                                             │
│ - FlowDroid 输出精确到方法签名（Soot 格式），节点必须与之对应                │
│ - Source/Sink 推断在 API 方法粒度（TelephonyManager.getDeviceId()）          │
│ - GraphRAG 查询可精确到 "query() 方法是否访问 SSN" 而非模糊的                │
│ "ContentProvider 是否泄漏"                                                   │
│ - 调用图边（DIRECT_CALL/VIRTUAL_CALL）天然是方法→方法                        │
│                                                                              │
│ 运行：python -c "from hpg.builder import build_hpg; build_hpg()"             │

│ def extract_method_sigs_from_smali(smali_dir: str) -> list[str]:             │
│     """从 Smali 提取 Soot 格式方法签名作为候选列表"""                        │
│     sigs = []                                                                │
│     class_pattern = re.compile(r'\.class .+ ([\w/]+);')                      │
│     method_pattern = re.compile(r'\.method .+? (\w+)\(([^)]*)\)([^{]+)')     │
│     for f in glob(f"{smali_dir}/edu/ksu/**/*.smali"):                        │
│         content = open(f).read()                                             │
│         cls_match = class_pattern.search(content)                            │
│         if not cls_match:                                                    │
│             continue                                                         │
│         java_class = cls_match.group(1).replace('/', '.')                    │
│         for m in method_pattern.finditer(content):                           │

│             sigs.append(f"<{java_class}: {ret} {name}({params})>")           │

  ---
  一、整体是什么

  这是一个Android APK 安全分析流水线，分析目标是一个存在路径权限漏洞的
  ContentProvider。

  整个系统的核心问题是：外部 App 能不能在没有权限的情况下，通过 ContentProvider 
  读取用户的 SSN（社保号）等敏感数据？

  答案是能，流水线的工作就是自动化地证明这件事，并把证据写进 Neo4j 图数据库。

  ---
  二、运行方式

  conda activate graph-vuls
  cd analysis-pipeline
  python main.py

  运行后会：
  1. 连接本地 Neo4j（Docker 容器）
  2. 依次触发 7 个分析节点
  3. 输出 output/report.md（Markdown 报告）和 output/state_dump.json

  ---
  三、数据在哪里

  claude-codes/
    analysis/smali/
      AndroidManifest.xml          ← 权限配置文件（分析起点）
      smali/edu/ksu/cs/benign/
        provider/UserDetailsContentProvider.smali  ← 有漏洞的组件
        MainActivity.smali
        UserDetailsActivity.smali
    android-platforms/android-27/  ← FlowDroid 需要的 Android SDK
    soot-infoflow-cmd-jar-*.jar    ← FlowDroid 可执行文件

  ---
  四、系统架构：7 个节点的 LangGraph 流水线

  manifest_agent
        ↓
  taint_agent

    [susi_confidence < 0.7 ?]──是──→ semantic_agent

                                     icc_bridge_node

                                    validation_agent

                                    report_generator

  每个节点接收共享的 AnalysisState，处理后返回更新的字段。Neo4j 
  是节点间的共享黑板——前一个节点写进去的东西，后一个节点会去读。

  ---
  IccTA 做的是什么

  IccTA 解决的是 Intent 链路上的污点传播：

  App A:
    startActivity(intent.putExtra("data", tainted_value))
          ↓ Intent ICC 链路
  App B:
    onCreate() → getIntent().getStringExtra("data") → sink

  IccTA 需要在代码里找到显式的 ICC 发送点（startActivity、sendBroadcast、startSe
  rvice），然后把发送端的污点连接到接收端的入口，继续跟踪传播。

  它的关键前提：ICC 链路在代码里是显式的。

  ---
  我们的 Layer B 做的是什么，以及为何不同

  这个 APK 的漏洞类型是 ContentProvider + 
  路径权限配置错误，它的攻击模型完全不同：

  攻击者 App（代码里什么都没有）：
    ContentResolver.query("content://edu.ksu.cs.benign.userdetails/", ...)
          ↓ 操作系统自动路由（不是 Intent）
  UserDetailsContentProvider:
    query() → 读 SSN CSV → MatrixCursor 返回给攻击者

  攻击者不需要发任何 Intent，也不需要引用目标 App 的任何代码。操作系统的
  ContentProvider 机制本身就是开放的——只要 URI 被访问，query() 就会执行。

  所以 IccTA 在这里找不到 ICC 链路，因为：
  1. 没有 startActivity / sendBroadcast 可以分析
  2. 攻击者 App 是未知的第三方，根本不在分析范围内
  3. 漏洞不在数据流的"传输过程"，而在"谁有权进来"这个配置上

  ---
  我们 Layer B 的本质：逻辑推断，而非污点传播

  IccTA 的跨组件分析是：跟着污点走（taint propagation across ICC links）

  我们的 Layer B 是：逻辑蕴含推断（logical entailment）

  if ContentProvider.exported == True
     and IntraPath exists (query() → sensitive data)
  then:
     任何外部 App 都能触发这条 IntraPath
     → 创建 CrossPath（置信度 0.95）

  这条推断规则的依据是 Android 安全语义，而不是程序数据流。它回答的是：

  "这个组件内的污点路径，在系统层面对外部 App 是否可达？"

  这是 可达性分析（Reachability）+ 权限语义推理，不是污点传播。

  ---
  更深的区别：检测的漏洞类别不同

  IccTA 检测的是数据流漏洞：敏感数据通过 Intent 意外泄漏到另一个组件。

  我们检测的是配置语义漏洞：权限配置的逻辑缺陷导致本该受保护的路径被绕过。

  具体来说，path-permission 只保护 /user 前缀的路径，但 ContentProvider 的根路径
   / 完全暴露——这是一个 Android 
  权限模型语义层面的漏洞，纯静态数据流分析天然无法表达"某个 URI
  路径是否受到权限保护"这个问题。

  这也是为什么我们需要 LLM 和 GraphRAG：

  Text2Cypher 查询："path-permission 配置能防止根路径被绕过吗？"
     ↓ 生成 Cypher
     MATCH (c:Component)-[:HAS_PATH_PERMISSION]->(pp:PathPermission)
     RETURN c.name, pp.pathPrefix, pp.readPermission
     ↓ 结果
     pathPrefix=/user  (根路径 / 没有保护)

  这种推理是 IccTA 和 FlowDroid 的静态规则引擎做不到的。

  ---
  总结一句话

  FlowDroid 问"数据在哪里流"，IccTA 问"数据能不能通过 Intent
  跨组件流"，我们问"外部 App 能不能绕过权限配置进来触发那条数据流"。三者是互补的
  ，不是重复的。我们的贡献在于用 Neo4j 图 + LLM 语义推理
  弥补了纯静态分析在权限语义理解上的盲区。
